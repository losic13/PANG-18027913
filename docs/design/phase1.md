# Phase 1 설계: 메인 화면 노출

[PLAN.md](../../PLAN.md)의 Phase 1 범위를 구현하기 위한 설계 문서다.

## 범위

- 앱 실행 시 메인 화면(타이틀)만 정적으로 렌더링한다.
- 게임 선택, 시작 버튼, 게임종료 등 상호작용 요소는 이후 Phase(2~4)에서 추가하며, Phase 1에서는 만들지 않는다.
- 즉, 이 단계의 "완성"은 클릭할 것도, 눌러볼 것도 없는 화면이 오류 없이 뜨는 것 자체다.

## 화면 구성 (참고: [main.md](../FEATURES/main.md))

- 화면 중앙(또는 상단)에 게임 타이틀 텍스트 "PANG" 표시
- 배경은 단순 단색 또는 기본 배경(디자인 디테일은 이후 폴리싱 Phase에서 다듬음)
- 이 시점에는 조작 안내 문구, 버전/미션 표시 등 부가 정보는 넣지 않는다(Phase 1은 타이틀만).

## 컴포넌트 구조

```
src/
  screens/
    MainScreen.tsx   # Phase 1에서 신규 추가: 타이틀만 렌더링하는 화면 컴포넌트
  App.tsx            # MainScreen을 렌더링하도록 교체
```

- `App.tsx`는 현재 `<h1>Hello World</h1>` (임시 배치 코드)를 제거하고 `<MainScreen />`을 렌더링하도록 변경한다.
- `MainScreen`은 이후 Phase에서 게임 선택 목록, 시작 버튼 등을 추가로 받을 자리이므로, 내부 구조를 처음부터 과도하게 나누지 않고 타이틀 하나만 담은 최소 컴포넌트로 시작한다.
- 화면 전환(메인 ↔ 게임)이 필요한 Phase 3부터 라우팅/상태 전환 방식을 결정하면 되므로, Phase 1에서는 별도 라우터나 화면 전환 상태를 도입하지 않는다.

## 스타일링

- 기존 `src/index.css`(전역 스타일)를 그대로 활용하고, `MainScreen` 전용 스타일이 필요하면 `src/screens/MainScreen.css`를 신규 추가한다.
- 레이아웃은 화면 전체를 채우는 flex 컨테이너로 타이틀을 중앙 정렬하는 정도로 단순하게 구성한다.

## 완료 조건 (Acceptance Criteria)

- [ ] `npm run dev`로 실행 시 흰 화면/콘솔 에러 없이 메인 화면이 표시된다.
- [ ] 화면에 "PANG" 타이틀 텍스트가 보인다.
- [ ] 버튼, 목록 등 클릭 가능한 요소가 전혀 없다(Phase 1 범위 밖 요소가 섞여 들어가지 않았는지 확인).
- [ ] `npm run build`, `npm run lint` 통과.

## Phase 2 이후를 위한 확장 지점

- `MainScreen` 컴포넌트는 이후 Phase 2(게임 선택 UI), Phase 3(시작 버튼)에서 타이틀 아래에 요소가 추가되는 방향으로 확장한다.
- 화면 전환(메인 → 게임 화면)은 Phase 3에서 논의하며, 별도 URL 라우팅 없이 컴포넌트 내부 상태로 처리할지 여부는 Phase 3 설계 시점에 결정한다.
